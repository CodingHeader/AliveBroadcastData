## 2026-05-21 需求讨论

### 确认的需求
1. 油猴脚本自动采集抖音来客"线索大屏"页面数据
2. 每天0点执行，采集前一天所有直播场次（每天2-3场，2-3个主播轮流）
3. 通过"上一场/下一场"按钮遍历场次
4. 每场采集6类数据：核心指标、复盘表、线索列表、评论、高意向用户、图片
5. 数据POST到服务端API，存入SQLite
6. 前台公开看板：数据展示、报告下载、趋势分析
7. 后台需登录（admin/admin123）：成单标记、主播配置、AI配置、邮箱配置
8. AI分析：每小时自动分析未分析场次，兼容OpenAI格式
9. 邮件推送：HTML格式，推送前日汇总+每场数据+AI分析
10. 报告生成：单场/日报/周报/月报，可筛选下载

### 确认的技术细节
- 按钮定位：`svg[data-log-name="上一场"]` / `svg[data-log-name="下一场"]`

- 场次时间格式："05-21 09：28～05-21 13：36"，以开播时间判断日期

- 成单=付款，人工后续标记

## 2026-06-05 修复首页浮点数精度问题

### 问题
首页"总消耗"显示为 `¥1234.6100000000001`，"线索平均成本"也可能出现同类问题。

### 根因
- 后端 `api.py:206` `func.sum(SessionMetric.ad_spend)` 返回浮点数，多次累加产生精度误差
- 后端 `api.py:311` `cur_spend`/`prev_spend` 未做 round 处理直接返回
- 前端 `dashboard.html:155` `'¥'+(spend||0)` 直接拼接，未做 toFixed 格式化
- 对比：同页 `lead_cost` 后端有 `round(...,1)`，历史消耗前端有 `.toFixed(0)`

### 修复
- 文件：`server/templates/dashboard.html`
- 改动：`'¥'+(spend||0)` → `'¥'+(spend||0).toFixed(2)`
- 改动：`'¥'+(leadCost||0)` → `'¥'+(leadCost||0).toFixed(2)`
- 方案A（前端格式化），保留后端原始精度便于后续计算

## 2026-05-22 油猴脚本"每日重试3次"机制实现（方案A）

### 需求
- 每天0点自动采集前一天的直播数据
- 采集完成后当天不再采集
- 若采集失败，最多自动重试3次，3次后彻底放弃，等待第二天0点循环执行

### 实现
1. **CONFIG 新增** `MAX_DAILY_ATTEMPTS: 3`（`user.js:28`）
2. **startCollection() 开头**增加尝试计数逻辑（`user.js:749-763`）：
   - 读取 `last_attempt_date` + `today_attempts`
   - 若 `today_attempts >= 3`，直接 return（当天静默）
   - 否则 `today_attempts++`，保存后进入采集流程
   - 关键点：**一进入函数就记录次数**，即使后续异常退出，次数也已计入
3. **startHeartbeat() 触发条件修改**（`user.js:838`）：
   - 原：`lastCollect !== today`
   - 新：`lastCollect !== today && todayAttempts < MAX_DAILY_ATTEMPTS`
   - 3次尝试耗尽后，心跳不再触发刷新
4. **次日自动重置**：第二天 `last_attempt_date !== today`，`today_attempts` 自动归零

### 行为效果
| 尝试次数 | 行为 |
|----------|------|
| 第1次 | 0点后首次触发 → 执行采集 |
| 第2次 | 若失败/异常 → 心跳继续触发 → 刷新后重试 |
| 第3次 | 若再失败 → 心跳最后一次触发 → 刷新后尝试 |
| 3次后 | startCollection() 开头直接跳过，心跳不再刷新，当天静默 |
| 第二天0点 | today_attempts 重置为0，重新开始 |

### 文件变更
- `tampermonkey/alive-broadcast-sync.user.js`：CONFIG + startCollection + startHeartbeat（3处修改）
- `.docs/使用说明.md`：5.2/5.3/5.4 节全面重写 + 配置项表格更新

## 2026-05-27 油猴脚本 v1.3.3 — 空数据入库修复

### 问题现象
凌晨自动采集时，指标完整率 `NaN% (0/0)`、线索0条、评论0条，空数据入库（session_id=9）。

### 根因分析
三个连锁Bug导致空数据穿透所有校验入库：
1. **SPA未就绪就采集**：`location.reload()` 后仅3秒固定等待，抖音来客SPA数据卡片未渲染（后台tab被Chrome节流）
2. **validateMetrics NaN穿透**：`metrics={}` → `(0-0)/0*100 = NaN` → `NaN < 70` 为 `false` → 校验通过
3. **last_collect_date无条件写入**：即使全部失败也标记"已完成"，心跳不再重试

### 修复内容（5处改动）

| 编号 | 改动 | 文件位置 |
|------|------|----------|
| A1 | `validateMetrics` NaN防护：`totalFields===0` 直接返回 false | `user.js:959-962` |
| A2 | 新增 `waitForPageReady(60s)` 轮询函数：等待 `[data-log-name]` 卡片出现 | `user.js:335-351` |
| A3 | 主入口 pending_collect 路径用 `waitForPageReady` 替代固定3秒 | `user.js:1448-1455` |
| A4 | `last_collect_date` 仅 `collected > 0` 时写入，允许心跳自然重试 | `user.js:1375-1377` |
| A5 | `sendToServer` 前检查 metrics 非全空，拒绝脏数据 | `user.js:1315,1327-1329` |

### 重试机制设计
- 无 `location.reload()` 循环，无额外状态变量
- 心跳（60秒间隔）自然提供有界重试，上限 = `MAX_DAILY_ATTEMPTS`(3)
- 采集窗口：凌晨1:00~7:00
- 全部失败 → `last_collect_date` 不写入 → 心跳下个周期重试

## 2026-06-09 修复"添加主播提示服务器内部错误"

### 问题

### 根因
`database.py` 中 `gender` 和 `style` 列的迁移代码缺少 `conn.commit()`，导致这两列从未被添加到数据库。

- `anchors` 表实际只有 `id, name, created_at, is_parttime` 四列
- `gender` 和 `style` 列缺失
- 后端 `create_anchor` 写入 `gender`/`style` 字段时，SQL 报错 → 500

对比：`is_parttime` 的迁移代码有 `conn.commit()`，所以该列成功添加。

### 修复内容

**文件1：`server/database.py`**
- 第37-40行：`gender` 迁移添加 `conn.commit()`
- 第41-44行：`style` 迁移添加 `conn.commit()`
- 第29-33行：`share_rate` 迁移添加 `conn.commit()`（同类问题一并修复）

**文件2：`server/routers/admin.py`**
- `create_anchor` 函数：添加 try-except，捕获 IntegrityError（重名）返回 `{"code":400, "message":"主播名称已存在"}`，通用异常返回 `{"code":500, "message":"添加失败: ..."}`
- `update_anchor` 函数：同理添加 try-except + rollback

**数据修复**：手动执行 SQL 添加 `gender` 和 `style` 列到已有数据库

### 验证结果

- Playwright 浏览器内 API 调用：添加+重名均通过 ✅

### 经验教训
- SQLAlchemy 2.0 的 `engine.connect()` 默认在事务中执行，DDL 语句（ALTER TABLE）也需要显式 `conn.commit()`

## 2026-06-23 修复侧边栏SPA导航高亮状态不一致（终极修复）

### 问题现象（两次修复）
1. 第一次修复后：点击菜单 A → 高亮仍停留在菜单 B（页面内容已切换，但侧边栏高亮未更新）
2. 第二次修复后（用户再次报错）：点击菜单 X → 高亮到错误的菜单 Y

### 根因分析（逐项排查）

**根因1：`Alpine.initTree` 会重置组件状态而非刷新**
- 对 `<aside>` 调用 `Alpine.initTree()` 不会重新计算已有 Alpine 组件的模板表达式，反而可能干扰内部状态追踪
- 正确做法：依赖 Alpine 的响应式属性赋值（`this._path = url`）来触发模板重算

**根因2：`get active()` getter 的依赖追踪不可靠**
- Alpine v3 在模板中调用 `isActive('/admin/leads')` 时，函数内部的 `this.active` → `this._path` 的间接引用可能丢失响应式追踪
- 正确做法：`isActive` 和 `_autoExpand` 直接读取 `this._path`，不经过 getter

**根因3：路径更新与异步加载的时序问题**
- 旧代码：`loadContent(url)` 异步 fetch → `this._path = url`（在 fetch 之后设置，延迟高）
- 新代码：`navigateTo(url)` 先同步设置 `this._path = url` → 再异步 `loadContent(url)`，确保 Alpine 立即响应

### 最终修复方案（2026-06-23 第三次修复，浏览器验证通过）

**根因诊断（前两次修复失败的原因）：**

| # | 问题 | 现象 |
|---|------|------|
| 1 | `<main>` 被包裹在 `x-data="sidebarApp()"` 内部 | `this._path = url` 触发 Alpine 重渲染时，`<main>` 被恢复为旧模板内容，刚 fetch 加载的新内容被覆盖 → "点第一次没反应" |
| 2 | `get active()` getter 间接引用 `_path` | Alpine 模板中 `isActive()` 内通过 `this.active` → `this._path` 的间接引用可能丢失响应式追踪 |
| 3 | `Alpine.initTree` 用于刷新已有组件 | `initTree` 只应用来初始化新 DOM，对已有组件调用会干扰内部状态 |

**最终修复（浏览器 CDP 自动化测试 5/5 通过）：**

| 改动 | 说明 |
|------|------|
| **`x-data="sidebarApp()"` 移到 `<aside>` 上** | `<main>` 不再是 sidebarApp 的子节点，Alpine 重渲染 sidebar 不会影响 main 内容 |
| **删除 `get active()` getter** | `isActive()` 和 `_autoExpand()` 直接读 `this._path`，不经过 getter |
| **删除 `Alpine.initTree` 侧边栏调用** | 不再手动刷新 DOM，依赖 Alpine 响应式系统 |
| **新增 `navigateTo(url)` 方法** | 统一导航入口：先 `this._path = url` 同步更新，再 `loadContent(url)` 异步加载 |
| **17 个侧边栏 `@click` 改为 `navigateTo`** | 点击 → 立即可见高亮变化，再等页面加载 |
| **popstate 也更新 `_path`** | 浏览器前进/后退时同步更新高亮和展开状态 |

### 经验教训（写入记忆，永久避免再犯）
1. **`<main>` 内容区必须独立于 sidebar 的 `x-data` 作用域**，否则任何 Alpine 重渲染都会覆盖 SPA 加载的内容
2. **`Alpine.initTree` 只用于新 DOM 初始化**，不能用于刷新已有组件。刷新应通过修改响应式属性来触发
3. **getter 在 Alpine 模板表达式中的响应式追踪不可靠**，函数内间接访问响应式属性可能导致依赖丢失。直接访问 `this._path` 更可靠
4. **路径更新必须在异步操作之前完成**（先 `this._path = url` 再 `await fetch`），确保用户点击后立即看到视觉反馈
5. **修复后必须用真实浏览器验证**，仅靠代码审查可能遗漏 Alpine 渲染时序问题
