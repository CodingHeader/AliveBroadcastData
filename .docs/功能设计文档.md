## 四、API接口设计

### 4.1 通用响应格式

**成功：** `{"code": 0, "message": "success", "data": {...}}`

**失败：** `{"code": 400, "message": "错误描述"}`

**分页：** `{"code": 0, "data": {"items": [...], "total": 100, "page": 1, "size": 20}}`

### 4.2 数据接收API

#### POST /api/session — 接收单场完整数据

请求体为 JSON，包含 start_time、end_time、metrics(44字段)、leads[]、comments[]、high_intent_users[]。完整 Schema 见 4.1 节数据流设计中的 JSON 示例。

**响应：** `{"code": 0, "message": "success", "session_id": 1}`

#### GET /api/session/check — 防重复检查

参数：`?start_time=2026-05-21 09:28:38`

响应（已存在）：`{"code": 0, "exists": true, "session_id": 1}`

响应（不存在）：`{"code": 0, "exists": false}`

## 五、数据流设计

### 5.1 采集→存储完整链路

```
油猴 DOM采集 → JSON构造 → fetch POST → FastAPI接收
  → Pydantic校验 → 解析时间/数值格式 → 计算duration
  → SQLAlchemy事务写入(sessions→metrics→leads→comments→high_intent)
  → 返回 {"code":0, "session_id":1}
```

### 5.2 数据解析规则

| 原始格式 | 解析结果 | 规则 |
|----------|----------|------|
| "34,180" | 34180 | 去逗号→int |
| "4.4万" | 44000 | "万"×10000→int |
| "1,061.58" | 1061.58 | 去逗号→float |
| "3.68%" | 0.0368 | 去%→float/100 |
| "30秒" | "30秒" | 保持文本 |
| "05-21 09：28" | "2026-05-21 09:28:00" | 全角：→半角:，补年份和秒 |

### 5.3 时间补全逻辑

油猴脚本中的时间补全：
1. 原始格式："05-21 09：28～05-21 13：36"
2. 全角"："替换为半角":"
3. 补充年份（使用当前年份，如果月份>当前月则用上一年）
4. 补充秒数":00"（如原始有秒则保留）
5. 结果："2026-05-21 09:28:00"

## 六、AI分析设计

### 6.1 默认系统提示词（后台可修改）

```
你是一位专业的直播数据分析师，擅长分析抖音本地生活直播间的运营数据。
请根据提供的直播数据，输出一份专业的分析报告。

分析要求：
1. 整体评价本场直播的表现（优秀/良好/一般/较差），给出综合评分（1-10分）
2. 分析流量效率（曝光进入率、观看时长、在线人数趋势）
3. 分析互动质量（评论率、点赞率、分享率、粉丝转化）
4. 分析转化效率（留资率、线索成本、组件点击率）
5. 分析线索质量（城市分布、留资路径偏好、停留时长分布）
6. 给出3-5条具体可执行的优化建议
7. 如果有多场数据对比，给出趋势判断

输出格式要求：使用Markdown格式，结构清晰，重点数据加粗。
```

### 6.2 默认用户提示词模板（可后台修改）
```
请分析以下直播数据：

## 基础信息
- 直播时间：{{start_time}} ~ {{end_time}}
- 直播时长：{{duration_minutes}}分钟
- 营销消耗：¥{{ad_spend}}

## 核心指标
{{metrics_text}}

## 线索列表（共{{leads_count}}条）
{{leads_text}}

## 评论明细（共{{comments_count}}条）
{{comments_text}}

## 高意向用户（共{{high_intent_count}}个）
{{high_intent_text}}

请给出完整分析报告。
```

### 6.3 周报/月报内容模板

**周报结构：**
1. 周度总览（场次数、总线索、总消耗、平均线索成本、成单数/金额、周ROI）
2. 环比对比表（本周 vs 上周，各指标变化率）
3. 每日数据表（日期、场次、线索、消耗、成本）
4. 主播表现（主播、场次、场均线索、场均成本）
5. AI周度分析（基于整周数据单独调用AI生成）

**月报结构：** 同周报，增加“月度趋势图描述”和“优化建议回顾”章节。

### 6.4 分析流程
```
1. 从数据库查询 analyzed=false 的场次
2. 加载该场次全量数据（指标+线索+评论+高意向）
3. 渲染用户提示词模板，填入数据
4. 调用OpenAI兼容API（system_prompt + user_prompt）
5. 将AI返回的Markdown存入 reports 表（type=session）
6. 更新 sessions 表 analyzed=true, analyzed_at=now
```

## 十、安全设计

| 项目 | 方案 |
|------|------|
| CORS | 允许油猴脚本跨域（配置 allowed_origins） |
| 前台API | 无需认证，公开访问 |

## 十一、DOM采集策略（油猴脚本）

### 11.1 Tab切换逻辑

| Tab名称 | 定位方式 | 默认可见 |
|----------|----------|----------|
| 核心指标 | 页面默认显示 | |
| 线索列表 | 页面默认显示 | |
| 评论明细 | `data-log-name="评论"` 点击切换 | |
| 高意向用户 | `data-log-name="高意向"` 点击切换 | |

采集顺序：核心指标 → 线索列表 → 点击评论Tab(等待500ms) → 采集评论 → 点击高意向Tab(等待500ms) → 采集高意向

### 11.2 选择器策略

- 优先使用 `data-log-name` 属性定位
- 其次使用可见文本内容匹配
- 不依赖动态class名（如 `V7dU-xxx`）
- 所有选择器集中定义在脚本顶部配置区

### 11.3 数据加载等待

- 切换场次后：MutationObserver 监听关键数据区域DOM变化
- 超时保护：最多等待30秒，超时跳过并记录日志
- 虚拟滚动：滚动到底部触发加载，检测列表长度不再变化时停止

## 十二、漏斗分析

### 12.1 转化漏斗定义

```
曝光次数 → 直播间观看次数 → >1分钟观看 → 互动人数 → 留资人数
```

### 12.2 各层转化率

| 层级 | 计算 |
|------|------|
| 曝光→观看 | 观看次数 / 曝光次数 |
| 观看→深度观看 | >1分钟观看 / 观看次数 |
| 深度观看→互动 | 互动人数 / >1分钟观看 |
| 互动→留资 | 留资人数 / 互动人数 |

在场次详情页和Dashboard中展示ECharts漏斗图。
