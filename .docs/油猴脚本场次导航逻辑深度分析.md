# 油猴脚本场次导航逻辑深度分析

## 执行摘要

基于代码事实的理性、辩证分析，当前实现**逻辑正确但存在效率问题**。用户提出的参考方案（获取当前场次→比较→翻页）与当前实现**本质相同**，但参考方案的表述更清晰。建议采用**改进方案**，在保持逻辑正确的前提下，优化效率和鲁棒性。

## 维度1：当前实现的正确性

### 1.1 逻辑流程（代码L776-818）

```javascript
async function navigateToDate(targetDate) {
    for (let i = 0; i < 50; i++) {
        const timeInfo = parseSessionTime(getTimeRangeText());  // 读取当前场次时间
        if (timeInfo.isLive) {
            // 直播中场次：点"上一场"跳过
            clickButton(SELECTORS.prevButton);
            continue;
        }
        if (timeInfo.startDate === targetDate) {
            return true;  // 已定位到目标日期
        }
        if (timeInfo.startDate > targetDate) {
            clickButton(SELECTORS.prevButton);  // 当前日期晚于目标 → 点"上一场"
        } else {
            clickButton(SELECTORS.nextButton);  // 当前日期早于目标 → 点"下一场"
        }
        // 边界检测：beforeText === afterText 判断场次是否切换
    }
}
```

**正确性评估：✅ 逻辑正确**

- 比较逻辑：`startDate > targetDate` → 点"上一场"；`startDate < targetDate` → 点"下一场"
- 这与用户参考方案**完全一致**（参考方案用"早于/晚于"的表述，当前实现用">/< 比较"）
- 边界检测：`beforeText === afterText` 判断场次是否切换，防止死循环
- 直播中场次处理：跳过正在直播的场次，继续往前翻

**能否处理"当前场次在1-几天前"的情况：✅ 能**

- 假设当前显示的是"05-20"的场次，目标是"05-21"
- 第1次循环：`startDate(05-20) < targetDate(05-21)` → 点"下一场"
- 第2次循环：`startDate(05-21) === targetDate(05-21)` → 返回true
- 最坏情况：需要翻N次页才能到达目标日期（N = 目标日期与当前日期的场次差）

**边界检测可靠性：✅ 可靠**

- `beforeText === afterText` 检测场次是否切换
- 如果按钮禁用或已到边界，场次不会改变，循环退出
- 防止了无限循环

## 维度2：效率问题

### 2.1 最坏情况分析

**场景：** 当前显示"05-20 09:00"，目标是"05-21"，该日有3场直播

| 循环次 | 当前场次 | 操作 | HTTP请求 | 说明 |
|--------|----------|------|----------|------|
| 1 | 05-20 09:00 | 点"下一场" | 1 | startDate < targetDate |
| 2 | 05-20 13:00 | 点"下一场" | 1 | startDate < targetDate |
| 3 | 05-20 18:00 | 点"下一场" | 1 | startDate < targetDate |
| 4 | 05-21 09:00 | 采集 | 1 | startDate === targetDate，开始采集 |
| 5 | 05-21 13:00 | 采集 | 1 | 点"下一场"后 |
| 6 | 05-21 18:00 | 采集 | 1 | 点"下一场"后 |
| 7 | 05-22 09:00 | 停止 | 0 | startDate > targetDate，退出 |

**总HTTP请求数：** 6次（3次定位 + 3次采集）

**问题：** 
- 定位阶段的3次翻页都需要等待 `waitForDataLoad()` 完成（每次500ms-30s）
- 如果当前场次在"一周前"，需要翻7×3=21次页才能定位

### 2.2 无效翻页分析

**当前实现的无效翻页：** 无

- 每次翻页都是有目的的（要么往前，要么往后）
- 边界检测防止了重复翻页

**但存在的低效：**

- 定位阶段和采集阶段分离
- 定位完成后，第一场采集时还要再点一次"下一场"才能进入采集循环
- 如果采集日期跨度大（如COLLECT_DAYS=7），定位时间累积

### 2.3 重复导航分析

**当前实现：** 无重复导航

- 每个targetDate只调用一次 `navigateToDate(targetDate)`
- 采集完该日期的所有场次后，才进入下一个targetDate

## 维度3：鲁棒性问题

### 3.1 边界检测可靠性

**当前实现：** `beforeText === afterText` 判断场次是否切换

**可靠性评估：✅ 可靠**

- 原理：如果点击按钮后场次时间文本不变，说明已到边界或按钮禁用
- 覆盖场景：
  - ✅ 已到最早场次（"上一场"禁用）
  - ✅ 已到最新场次（"下一场"禁用）
  - ✅ 页面加载失败（文本不变）
  - ✅ 按钮禁用（点击无效）

**潜在风险：**

- ❌ 如果DOM加载延迟，`getTimeRangeText()` 可能返回旧值，导致误判
  - 当前有 `waitForDataLoad()` 保护，但超时时间为30秒，可能过长
  - 建议：增加重试机制，如果文本不变则再等待1秒后重试

### 3.2 死循环风险

**当前实现：** 最多循环50次（L776）

**死循环风险评估：✅ 低**

- 循环上限：50次
- 边界检测：`beforeText === afterText` 时退出
- 直播中场次处理：跳过后继续，不会卡住

**但存在的问题：**

- 如果 `getTimeRangeText()` 始终返回空字符串（DOM加载失败），会循环50次后超时
- 建议：增加连续失败计数，如果连续3次无法解析时间则退出

### 3.3 越界风险

**当前实现：** 无越界风险

- 按钮禁用检测：`isButtonDisabled()` 检查class
- 场次日期检测：`dateSet.has(timeInfo.startDate)` 检查是否在采集范围内
- 直播中场次处理：跳过

## 改进方案

### 问题清单

基于上述分析，当前实现存在以下可改进的地方：

| # | 问题 | 严重度 | 影响 |
|---|------|--------|------|
| 1 | 定位阶段效率低（需多次翻页等待） | 中 | 采集速度慢 |
| 2 | 连续DOM加载失败时无退出机制 | 低 | 可能循环50次 |
| 3 | 边界检测依赖文本比较，可能误判 | 低 | 极端情况下可能重复翻页 |
| 4 | 采集日期跨度大时定位时间累积 | 中 | 采集速度慢 |

### 改进方案A：二分查找定位（推荐）

**核心思想：** 利用场次时间的单调性，用二分查找加速定位

```javascript
async function navigateToDateBinary(targetDate) {
    // 第1步：确定搜索范围（最早和最新场次）
    const earliest = await findEarliestSession();  // 点"上一场"直到禁用
    const latest = await findLatestSession();      // 点"下一场"直到禁用
    
    // 第2步：二分查找目标日期
    let left = earliest, right = latest;
    while (left <= right) {
        const mid = (left + right) / 2;
        await navigateToSession(mid);
        const timeInfo = parseSessionTime(getTimeRangeText());
        if (timeInfo.startDate === targetDate) return true;
        if (timeInfo.startDate < targetDate) left = mid + 1;
        else right = mid - 1;
    }
    return false;
}
```

**优势：**
- ✅ 定位时间从O(N)降低到O(log N)
- ✅ 对于COLLECT_DAYS=7的场景，从21次翻页降低到5次

**劣势：**
- ❌ 实现复杂度高
- ❌ 需要额外的"查找最早/最新"函数
- ❌ 对于小规模数据（通常<10场/天）收益不明显

**建议：** 当前数据规模（2-3场/天）不需要二分查找，收益不足以抵消复杂度。

### 改进方案B：增强边界检测和重试（推荐）

**核心改进：**

1. **连续失败计数**
   ```javascript
   let consecutiveFailures = 0;
   if (beforeText === afterText) {
       consecutiveFailures++;
       if (consecutiveFailures >= 3) {
           log('连续3次无法切换场次，退出定位');
           return false;
       }
       await sleep(1000);  // 等待后重试
       continue;
   } else {
       consecutiveFailures = 0;
   }
   ```

2. **DOM加载失败检测**
   ```javascript
   if (!timeInfo.startDate) {
       log('无法解析场次时间，等待加载...');
       await sleep(2000);
       continue;
   }
   ```

3. **循环上限优化**
   ```javascript
   const maxIterations = CONFIG.COLLECT_DAYS * 10 + 20;  // 动态计算
   for (let i = 0; i < maxIterations; i++) { ... }
   ```

**优势：**
- ✅ 实现简单（仅需修改现有逻辑）
- ✅ 提高鲁棒性
- ✅ 无性能损失

**劣势：**
- ❌ 效率提升有限

**建议：** ✅ **采用此方案**

### 改进方案C：缓存场次列表（可选）

**核心思想：** 脚本启动时缓存所有场次的时间列表，后续定位时直接查表

```javascript
async function cacheAllSessions() {
    const sessions = [];
    // 点"上一场"直到禁用，记录所有场次时间
    while (!isButtonDisabled(SELECTORS.prevButton)) {
        sessions.push(parseSessionTime(getTimeRangeText()));
        clickButton(SELECTORS.prevButton);
        await waitForDataLoad();
    }
    GM_setValue('session_cache', JSON.stringify(sessions));
    return sessions;
}

async function navigateToDateCached(targetDate) {
    const sessions = JSON.parse(GM_getValue('session_cache', '[]'));
    const targetSession = sessions.find(s => s.startDate === targetDate);
    if (!targetSession) return false;
    // 直接导航到目标场次（需要实现"跳转到指定场次"的逻辑）
    return true;
}
```

**优势：**
- ✅ 定位时间最短（O(1)查表）
- ✅ 支持快速多次定位

**劣势：**
- ❌ 初始化时间长（需要遍历所有场次）
- ❌ 缓存可能过期（新场次上线时需要刷新）
- ❌ 无法"跳转到指定场次"（抖音来客页面不支持）

**建议：** ❌ **不采用**（收益不足以抵消复杂度）

## 最终建议

### 唯一最优方案：改进方案B（增强边界检测和重试）

**理由：**

1. **逻辑正确性：** 当前实现已正确，无需改变核心逻辑
2. **效率充分：** 当前效率（10-30秒/天）已可接受，无需二分查找
3. **鲁棒性提升：** 增强边界检测和重试机制，提高容错能力
4. **实现成本低：** 仅需修改现有代码，无需重构

### 具体改进清单

| 改进项 | 代码位置 | 修改内容 |
|--------|----------|----------|
| 1 | navigateToDate() | 增加连续失败计数，3次失败后退出 |
| 2 | navigateToDate() | 循环上限改为动态计算：`CONFIG.COLLECT_DAYS * 10 + 20` |
| 3 | waitForDataLoad() | 超时时间改为10秒（当前30秒过长） |
| 4 | startCollection() | 增加采集失败计数，失败>0时发送告警 |

### 预期效果

- ✅ 鲁棒性提升：连续失败时能正确退出，避免无限循环
- ✅ 用户体验改善：采集失败时有明确告警
- ✅ 维护成本低：改动最小化
