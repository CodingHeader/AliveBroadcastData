### 第二轮修复（2026-06-11 三场景Bug修复）

#### Bug① 临时加班存不进（高严重度）
- **根因**：`schedule_log.html` `saveOvertime()` 发送的 slots 没有 `isNew: true` 标记，后端 `admin.py` 只处理 `isNew=true` 的 slot → `overtime_data` 恒为空数组
- **修复**：`saveOvertime()` 中新增 `isNew: newSlots.some(...)` 判断，仅新加班时段标记为 true

#### Bug② 排班日志不显示加班时段（中严重度）
- **根因**：`roomSlots()` 只过滤 `logData.slots`（ScheduleSlot），未合并 `logData.overtime_slots`（BindingSlot）
- **修复**：`roomSlots()` 合并两个数组，同time_slot的加班时段覆盖计划时段，不在计划中的追加

#### Bug③ 加班不生成SessionAnchor不进薪酬（中严重度）
- **根因**：`_sync_session_anchors()` 只查询 `ScheduleSlot`，不读 `BindingSlot`；且 `update_binding_slots` 保存加班后不触发同步
- **修复**：`_sync_session_anchors` 新增 `binding_id` 参数，同步查询 `BindingSlot`；`update_binding_slots` 保存后调用同步

#### Bug④⑤ 配置明细保存后重开不显示已有主播（中严重度）
- **根因④**：`saveSlots()` 中 `anchor_slot: s.anchor_slot || null` 把 falsy 值(如"")转为 null 存入DB
- **根因⑤**：`getSlotData()` 自动修复逻辑 `if(slot_status=='on_air_anchor' && !anchor_slot)` 把 null 覆盖为主播1
- **修复**：删除 `getSlotData` 的自动修复；移除失效的 `anchorParttime` 兼职标记（方案层无映射）

## 2026-06-11 直播账户管理界面修复+首次进入默认页

### 问题
1. 直播账户管理界面没有账户数据
2. "新增"表单在最外层，没有在弹出框中展示
3. 首次进入后台时应默认显示排班日历，所有一级菜单缩起

### 根因分析（3个问题）
1. **API 500错误 — ad_accounts.account_id NOT NULL约束** — 数据库中account_id列为NOT NULL，但模型定义为nullable=True。创建账户时未提供account_id导致IntegrityError。这是历史遗留问题：建表时account_id没有nullable=True，后续模型修改了但SQLite不支持ALTER COLUMN
2. **弹窗不关闭 — Alpine v3响应式被破坏** — `openCreate()`中`this.form = { account_name: '', ... }`创建新对象替换了Alpine Proxy代理的form对象，后续x-model绑定到新对象但addAccount()读取的this.form可能指向旧Proxy。同时`style="display:none"`与Alpine v3的x-show交互问题
3. **首次进入默认页** — login.html跳转地址已改为/admin/schedule-calendar，sidebarApp初始化逻辑已处理首次进入全缩起

### 修复
1. **admin.py create_ad_account** — 如果未提供account_id，自动生成`auto_{timestamp}`格式的唯一值，避免NOT NULL约束失败
2. **ad_accounts.html openCreate()** — 将`this.form = {...}`改为逐属性重置`this.form.account_name = ''`等，避免破坏Alpine v3 Proxy响应式
3. **ad_accounts.html 弹窗** — 去掉`style="display:none"`，改用`x-cloak`+`x-transition.opacity`，避免与Alpine v3 x-show交互问题
4. **admin.py** — 添加`import time`

### 修改文件清单
- `server/routers/admin.py` — create_ad_account自动生成account_id + import time
- `server/templates/admin/ad_accounts.html` — openCreate逐属性重置form + 弹窗去掉style="display:none"

## 2026-06-11 后台菜单栏+招生团队页面修复

### 问题
1. 菜单栏点击没反应，反复点击才勉强能用
2. 招生团队界面异常，菜单栏点击后内容页没数据

### 根因分析（4个问题）
1. **Alpine v3响应式被破坏** — `toggleExpand`和`_autoExpand`中使用了`this.expand = {...this.expand}`重新赋值整个对象，破坏了Alpine v3的Proxy响应式代理，导致`x-show`不更新DOM
2. **SPA导航正则匹配嵌套main标签** — `loadContent`使用`/<main[^>]*>([\s\S]*?)<\/main>/i`非贪婪匹配，在嵌套main标签结构中匹配到错误的位置，导致sidebar被重复插入或内容不完整
3. **3个页面script在block scripts中** — recruit_teams.html、ad_plans.html、leads.html的`<script>`放在`{% block scripts %}`中，该块在`<main>`标签外，SPA导航时script不会被执行
4. **_autoExpand在init中修改expand但Alpine初始渲染可能未完成** — 导致首次加载时x-show状态与数据不一致

### 修复
1. **base.html sidebarApp** — 去掉`this.expand = {...this.expand}`，改为在函数返回前预先计算expand初始值（合并localStorage+当前路径），避免在init中修改expand
2. **base.html loadContent** — 改为匹配`class含flex-1和bg-gray-50`的admin main标签，使用`document.querySelector('main.flex-1')`定位正确的DOM元素
3. **recruit_teams.html/ad_plans.html/leads.html** — 将`<script>`从`{% block scripts %}`移到`{% block admin_content %}`内部
4. **loadContent去掉同路径跳过** — 移除`if (!isPopState && url === location.pathname) return`限制

### 修改文件清单
- `server/templates/admin/base.html` — sidebarApp初始化逻辑+loadContent正则+DOM定位
- `server/templates/admin/recruit_teams.html` — script移入admin_content
- `server/templates/admin/ad_plans.html` — script移入admin_content

### 需求
1. poll-now 接口报500错误（原因：venv_linux缺失`requests`模块）
2. leads页面按留资时间倒序+分页UI
3. 采集后没有钉钉推送（原因：推送由调度器单独管理，与采集解耦）
4. 调度器自动轮询不产生日志（根因：旧进程启动时requests未安装+`--reload`导致worker频繁重启）

### 诊断结论
- **500根因**：`venv_linux`的Python中未安装`requests`，`clue_service.py`导入`requests`时报`ModuleNotFoundError`
- **无推送原因**：采集与推送已于206-06-11解耦，推送由调度器每10分钟执行`_check_unassigned_clues()`完成，非实时触发
- **auto-polling不工作根因**：
  1. 旧server进程（PID 859887, 启动时间18:23）启动时`requests`未安装，`start_scheduler()`静默失败
  2. 安装`requests`后，`--reload`模式持续检测到目录变化，worker进程频繁重启（每2秒一次），scheduler刚启动又被关闭，无法执行30秒周期任务
  3. 旧进程的`server.log`写入`server/server.log`导致watchfiles持续检测"变化"

### 修复
1. 安装`requests==2.34.2`到`venv_linux`
2. 杀死旧进程(PID 859887)，不带`--reload`重启调度器（使用`venv_linux/bin/python -c "import uvicorn; uvicorn.run(...)"`）
3. leads.html添加分页UI（页数/每页条数/总条数/上一页/下一页）+ 排序改为`lead_time DESC`
4. api.py线索列表查询排序从`created_at DESC`改为`lead_time DESC, created_at DESC`

## 2026-06-11 排班方案/排班日历首次加载无数据修复

### 根因
SPA导航 `loadContent()` 中 `adminMain.innerHTML = mainMatch[1]` → Alpine MutationObserver 异步检测到旧组件被移除并计划清理 → 紧接着同步调用 `Alpine.initTree(adminMain)` → MO清理尚未执行，Alpine内部可能有旧组件残留引用 → 新组件被跳过初始化 → `x-init="loadData()"` 不触发 → `this.plans` 保持 `[]`。

第二次点击时 MO已完成清理，DOM干净，initTree正常。

### 修复（方案A）
`server/templates/admin/base.html:82` — 在 `Alpine.initTree(adminMain)` 前插入：
```js
await new Promise(resolve => setTimeout(resolve, 0));
```
让 Alpine MutationObserver 在当前宏任务结束后、微任务队列清空后先完成旧组件清理，再执行 initTree。

### 效果
- 消除首次 SPA 导航时 Alpine MO 清理与 initTree 的竞态
- Jinja2 自动 reload 模板，无需重启服务器
